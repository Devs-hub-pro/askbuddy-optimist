# Phase 1.1 Cleanup 执行顺序

日期：2026-04-22  
约束：不新增 Pack、不做大 cleanup、不做大前端重构

## 先做（建议试运行前或试运行早期完成）

## 1) smoke 执行稳定性收口
- 内容：统一 `test:smoke` 的环境变量读取方式，减少人工注入依赖。
- 影响：提升日常回归稳定性，降低误报。
- 是否影响试运行：**建议优先做**（低风险高收益）。

## 2) 高频类型层小范围收口
- 内容：更新高频模块 types（Profile/MyEarnings/Discover/Search），去除明显旧字段残留。
- 影响：减少查询层歧义与误用。
- 是否影响试运行：**建议优先做**（不改业务行为）。

## 3) 关键 RPC 轻量回归脚本固化
- 内容：把 `accept_answer_v2`、`create_system_notification_v2`、`transition_order_status_v2` 抽检脚本纳入日常回归流程。
- 影响：提高 Freeze 期回归效率。
- 是否影响试运行：**建议优先做**（不改功能）。

## 后做（建议观察一版后再动）

## 4) legacy fallback 退役
- 内容：基于命中数据逐步退役 `user_followers`、posts/comments legacy fallback。
- 影响：涉及兼容层，需谨慎。
- 是否影响试运行：**不影响，建议观察后再做**。

## 5) 低频旧 trigger/policy 清理
- 内容：清理重复/低频历史策略与触发器，统一命名。
- 影响：数据库层风险高于查询层，需分批。
- 是否影响试运行：**不影响，建议后置**。

## 6) 展示层命名统一（Discover 等）
- 内容：逐步把展示层 legacy 命名替换为新命名。
- 影响：前端表现层改动，虽然不大但应避开试运行关键窗口。
- 是否影响试运行：**不影响，建议后置**。

## 执行原则（必须遵守）

- 每次仅处理一个小主题，避免并行过多变更。
- 每个任务必须可回滚、可审阅、可验证。
- 任何任务若触发主链路风险，立即降级为观察项，不强推上线前完成。
